Anchor — Project Brief

A consumer web app helping cancer patients navigate administrative chaos after diagnosis.

1. The Problem

A cancer diagnosis doesn't just bring medical challenges — it triggers an overwhelming wave of administrative tasks that patients are completely unprepared for:

  • Understanding and preserving health insurance (COBRA deadlines, appeals)
  • Navigating employment rights (FMLA, short-term disability, EAP)
  • Finding financial assistance (government programs, pharma patient assistance, nonprofit grants)
  • Managing critical deadlines that can't be missed
  • Generating legal and medical documents (appeal letters, leave requests)

Most patients figure this out alone, often missing benefits they're entitled to and deadlines that cost them thousands of dollars.

2. The Product — What Anchor Does

Anchor is a personalized administrative hub for cancer patients. After a short onboarding, users get:

Feature What it does
Action Checklist Prioritized list of things to do (urgent → low), categorized by insurance, employment, financial, legal, and medical
Benefits Finder AI-generated list of government programs, employer benefits, pharmaceutical assistance, and nonprofit grants the user likely qualifies for
Document Generator AI-generated documents pre-filled with the user's information: FMLA request, insurance appeal letter, COBRA notice, disability request, hardship letter, medical records request
Deadline Tracker Timeline of critical dates with urgency badges — COBRA windows, appeal deadlines, enrollment periods

3. Business Model

  • Price: $5/month flat subscription
  • Payment processor: Stripe (subscription billing)
  • Users can complete onboarding for free, but need an active subscription to access the dashboard features
  • Subscription status is stored in Supabase and checked on every dashboard route via the proxy (middleware)

7. Database Schema (Supabase / PostgreSQL)

TableKey columns
profiles id (ref auth.users), name, cancer_type, stage, diagnosis_date, employer_name, employer_size, insurance_type, insurance_provider, treatment_status, stripe_customer_id, subscription_status
action_items id, user_id, title, description, category (insurance|employment|financial|legal|medical), priority (urgent|high|medium|low), status (pending|in_progress|completed|skipped), due_date
benefits id, user_id, name, type (government|employer|nonprofit|pharmaceutical), description, eligibility_met, application_url, status (not_started|applied|approved|denied)
documents id, user_id, document_type, title, content (full text of generated document)
deadlines id, user_id, title, date, description, action_item_id, is_completed

All tables have Row Level Security enabled — users can only access their own rows. A trigger auto-creates a profile row when a user signs up.

8. Onboarding Flow

4-step form that collects:

  1. Diagnosis: cancer type, stage, diagnosis date, treatment status
  2. Employer: employer name, employer size (determines FMLA eligibility)
  3. Insurance: insurance type, insurance provider
  4. Review: confirm everything, then save to Supabase profiles table

After onboarding, users are redirected to /dashboard/upgrade to subscribe.

9. Subscription & Auth Flow

  1. User signs up → Supabase creates auth user + profile row
  2. User completes onboarding → profile fields saved
  3. User subscribes via Stripe Checkout → Stripe sends webhook to /api/stripe/webhook
  4. Webhook updates profiles.subscription_status to active
  5. proxy.ts checks subscription on every /dashboard/* request — redirects to /dashboard/upgrade if not active

Important: The Stripe webhook secret (STRIPE_WEBHOOK_SECRET) can only be set up after the app is deployed to Vercel, because Stripe needs a live public URL. Until then, subscription status must be manually set to active in the Supabase Table Editor for testing.

13. Known Issues / Notes

  • Supabase email confirmation should be disabled during development (Auth → Providers → Email → turn off "Confirm email") — otherwise onboarding saves fail silently because the user isn't fully authenticated yet
  • Subscription status must be manually set to active in Supabase Table Editor during local testing (since webhook needs a live URL)
  • The app uses Next.js 16 which renamed middleware.ts to proxy.ts — this is already handled correctly
  • Stripe and Anthropic clients use lazy initialization so they don't throw errors at build time when env vars aren't set
